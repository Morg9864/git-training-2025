SSH vs HTTPS: Quelle méthode choisir ?

🔐 SSH (recommandé)

  • Avantages:
  • ✅ Pas de mot de passe à chaque push
  • ✅ Plus sécurisé
  • ✅ Authentification par clé
  • Inconvénient:
  • ⚠️ Configuration initiale nécessaire

🌐 HTTPS

  • Avantages:
  • ✅ Simple et rapide
  • ✅ Fonctionne partout
  • Inconvénients:
  • ⚠️ Demande identifiants régulièrement
  • ⚠️ Moins pratique au quotidien

🔑 Générer une clé SSH

# Générer la clé (appuyer sur Entrée pour valider)
ssh-keygen -t ed25519 -C "email@example.com"

# Copier la clé publique
cat ~/.ssh/id_ed25519.pub

Puis: GitHub → Settings → SSH and GPG keys → New SSH key

Commandes de gestion des branches

Créer et naviguer

# Voir toutes les branches
git branch

# Créer une nouvelle branche
git branch feature/nouvelle-fonction

# Créer et basculer en une commande (méthode 1)
git checkout -b feature/nouvelle-fonction

# Créer et basculer (méthode 2, plus moderne)
git switch -c feature/nouvelle-fonction

# Basculer vers une branche existante
git switch main

Fusionner et supprimer

# Fusionner une branche dans la branche actuelle
git merge feature/nouvelle-fonction

# Supprimer une branche locale (après merge)
git branch -d feature/nouvelle-fonction

# Forcer la suppression (même non mergée)
git branch -D feature/non-utilisee

Résolution des conflits

❓ Qu'est-ce qu'un conflit Git ?

Un conflit survient quand Git ne peut pas automatiquement fusionner des modifications sur les mêmes lignes de code dans des branches différentes.

🔥 Situations courantes de conflits

Merge de branches

Deux développeurs modifient le même fichier au même endroit

Pull avec modifs locales

Vous avez modifié localement ce qui a été modifié à distance

⚠️ Important: Les conflits sont normaux et font partie du workflow collaboratif !
15/36

Identifier et comprendre les conflits

🔍 Détecter les fichiers en conflit

git status
both modified: app.js
both modified: style.css

📝 Marqueurs de conflit

<<<<<<< HEAD (votre version actuelle)
const titre = "Mon Application";
=======
const titre = "Application Web";
>>>>>>> feature/nouveau-titre
💡 Lecture des marqueurs:
• <<<<<<< = Début de votre version (HEAD)
• ======= = Séparateur
• >>>>>>> = Fin de la version entrante
16/36

3 options pour résoudre un conflit

1️⃣ Version locale

Garder VOTRE version

git checkout --ours fichier.js

2️⃣ Version distante

Garder la version ENTRANTE

git checkout --theirs fichier.js

3️⃣ Fusion manuelle

Combiner les deux (recommandé)

Éditer le fichier manuellement

✅ Workflow de résolution complet

# 1. Identifier les conflits
git status

# 2. Ouvrir et éditer les fichiers en conflit
code app.js

# 3. Retirer les marqueurs et choisir la version finale
# 4. Ajouter les fichiers résolus
git add app.js

# 5. Finaliser le merge
git commit -m "resolve: conflit résolu sur app.js"

# 6. Pousser
git push

Pull Requests sur GitHub

🔀 Qu'est-ce qu'une Pull Request (PR) ?

Une Pull Request est une demande de fusion de votre branche dans une autre (généralement main). C'est le cœur du workflow collaboratif sur GitHub.

✨ Avantages des Pull Requests

👀

Revue de code

L'équipe peut relire et commenter

💬

Discussion

Échanger sur les changements

📊

Traçabilité

Historique des décisions

✅

Validation

Tests automatiques CI/CD

Cliquer sur "Create pull request"

Méthodes de fusion (Merge) sur GitHub

🔀 Merge commit

Par défaut

  • ✅ Garde tous les commits
  • ✅ Historique complet
  • ⚠️ Peut être verbeux
Recommandé pour features importantes

📦 Squash and merge

Regrouper commits

  • ✅ Historique propre
  • ✅ 1 commit par PR
  • ⚠️ Perd détails intermédiaires
Idéal pour petites PR

⚡ Rebase and merge

Linéariser l'historique

  • ✅ Historique linéaire
  • ✅ Pas de merge commit
  • ⚠️ Réécrit l'historique
Pour experts
💡 Conseil: Utilisez "Squash and merge" pour 90% des cas

Pull Requests de qualité

📏 Taille idéale d'une PR

50-200 lignes

Plus facile à reviewer, moins d'erreurs manquées

✅ Checklist d'une bonne PR

  • Titre descriptif: Résume le changement en une phrase
  • Description complète: Pourquoi, quoi, comment
  • Screenshots/GIFs: Si changements visuels
  • Tests ajoutés: Couvrir le nouveau code
  • Documentation: README mis à jour si nécessaire
  • Pas de conflits: Résoudre avant de soumettre
💡 Template de description PR: Utilisez un template avec sections "Changements", "Tests", "Screenshots", "Checklist"
23/36

Revue de code bienveillante

✅ Bonnes pratiques reviewer

  • Être constructif et respectueux
  • Expliquer le "pourquoi"
  • Suggérer des améliorations
  • Approuver si satisfait
  • Répondre rapidement (< 24h)

✅ Bonnes pratiques auteur

  • Accepter la critique positivement
  • Répondre à tous les commentaires
  • Faire les modifications demandées
  • Remercier les reviewers
  • Auto-review avant de soumettre

💬 Exemples de commentaires constructifs

✅ Bien: "Bonne idée ! On pourrait simplifier cette fonction en utilisant map() plutôt qu'une boucle for. Qu'en penses-tu ?"
❌ À éviter: "Ce code est mauvais. Refais-le."

Glossaire - Historique & Annulation

Historique

git log Voir l'historique
git log --oneline Historique condensé
git log --graph Historique graphique
git show <commit> Voir un commit
git blame <fichier> Qui a modifié quoi

Annulation

git reset <fichier> Unstage un fichier
git reset --hard Annuler tout (dangereux)
git revert <commit> Annuler un commit
git restore <fichier> Restaurer un fichier
git stash Mettre de côté
⚠️ Attention: git reset --hard supprime définitivement les modifications non commitées !
