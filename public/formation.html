Introduction à Git, GitHub et GitLab

Git

Système de contrôle de version distribué

  • Créé en 2005 par Linus Torvalds
  • Fonctionne en local
  • Open source et gratuit
GitHub

Plateforme d'hébergement n°1

  • Propriété de Microsoft
  • 100M+ de développeurs
  • Leader du marché
GitLab

Alternative complète

  • Open source (CE)
  • Auto-hébergeable
  • CI/CD intégré nativement
Différence clé : Git = outil local  |  GitHub / GitLab = plateformes cloud

SSH vs HTTPS : Quelle méthode choisir ?

SSH — Recommandé
  • Pas de mot de passe à chaque push
  • Plus sécurisé
  • Authentification par clé cryptographique
  • Configuration initiale nécessaire
HTTPS
  • Simple et rapide à démarrer
  • Fonctionne derrière les firewalls
  • Demande identifiants régulièrement
  • Moins pratique au quotidien
Générer une clé SSH
# Générer la clé (appuyer sur Entrée pour valider)
ssh-keygen -t ed25519 -C "email@example.com"

# Afficher la clé publique à copier sur GitHub
cat ~/.ssh/id_ed25519.pub

Puis : GitHub → Settings → SSH and GPG keys → New SSH key

Commandes de branches — Créer & naviguer

# Voir toutes les branches
git branch

# Créer une nouvelle branche
git branch feature/nouvelle-fonction

# Créer et basculer en une commande (méthode moderne)
git switch -c feature/nouvelle-fonction

# Basculer vers une branche existante
git switch main

# Méthode classique (toujours valide)
git checkout -b feature/nouvelle-fonction
14 / 47

Commandes de branches — Fusionner & supprimer

# Fusionner une branche dans la branche actuelle
git merge feature/nouvelle-fonction

# Supprimer une branche locale (après merge)
git branch -d feature/nouvelle-fonction

# Forcer la suppression (même non mergée)
git branch -D feature/non-utilisee

# Pousser une branche vers le dépôt distant
git push -u origin feature/nouvelle-fonction
Bonne pratique : Supprimez les branches après leur merge pour garder le dépôt propre.

Résolution des conflits

Un conflit survient quand Git ne peut pas fusionner automatiquement des modifications sur les mêmes lignes de code dans des branches différentes.

Merge de branches

Deux développeurs modifient le même fichier au même endroit

Pull avec modifs locales

Vous avez modifié localement ce qui a été modifié à distance

Important : Les conflits sont normaux et font partie du workflow collaboratif !

Identifier et lire les conflits

Détecter les fichiers en conflit

both modified: app.js
both modified: style.css
Marqueurs dans le fichier
<<<<<<< HEAD (votre version actuelle)
const titre = "Mon Application";
=======
const titre = "Application Web";
>>>>>>> feature/nouveau-titre
Lecture : <<<<<<< Votre version (HEAD)  ·  ======= Séparateur  ·  >>>>>>> Version entrante

3 options pour résoudre un conflit

Option 1 Garder votre version
git checkout --ours fichier.js
Option 2 Garder la version entrante
git checkout --theirs fichier.js
Option 3 — Recommandée Fusion manuelle

Éditer le fichier pour combiner les deux versions comme vous le souhaitez

Workflow complet de résolution

# 1. Identifier les fichiers en conflit
git status

# 2. Ouvrir et éditer le fichier
code app.js

# 3. Supprimer les marqueurs, choisir la version finale
# (édition manuelle dans l'éditeur)

# 4. Marquer le conflit comme résolu
git add app.js

# 5. Finaliser le merge
git commit -m "resolve: conflit résolu sur app.js"

# 6. Pousser
git push

Pull Requests sur GitHub

Une Pull Request (PR) est une demande de fusion de votre branche dans une autre (généralement main). C'est le cœur du workflow collaboratif sur GitHub.

Revue de code L'équipe relit et commente
Discussion Échanger sur les changements
Traçabilité Historique des décisions
Validation Tests automatiques CI/CD

Méthodes de fusion (Merge) sur GitHub

Merge commit Par défaut
  • Garde tous les commits
  • Historique complet
  • Peut être verbeux
Features importantes
Squash and merge Regrouper les commits
  • Historique propre
  • 1 commit par PR
  • Perd les détails intermédiaires
Petites PR
Rebase and merge Linéariser l'historique
  • Historique linéaire
  • Pas de merge commit
  • Réécrit l'historique
Pour experts
Conseil : Utilisez "Squash and merge" pour 90% des cas. Simple et propre.

Messages de commit — Bons vs mauvais exemples

Bons exemples
feat(auth): ajout du système de login OAuth
fix(api): correction du timeout sur la requête GET /users
docs(readme): mise à jour des instructions d'installation
refactor(utils): simplification de la fonction de validation
Mauvais exemples
update
fix stuff
WIP
changes
asdfgh
Règle d'or : Un commit = une chose. Si vous écrivez "et" dans votre message, séparez en deux commits.

Pull Requests de qualité

Taille idéale · Plus facile à reviewer, moins d'erreurs manquées

Checklist d'une bonne PR

  • Description complète : Pourquoi, quoi, comment
  • Screenshots / GIFs : Si changements visuels
  • Tests ajoutés : Couvrir le nouveau code
  • Pas de conflits : Résoudre avant de soumettre
